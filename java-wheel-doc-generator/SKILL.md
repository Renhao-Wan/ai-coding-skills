**触发关键词（用户输入任意之一即可激活）：**  
写项目文档、分析这个轮子、生成starter文档、分析设计思路、完善DESIGN_DOC、写设计文档、分析扩展点等相关关键词。

**AI 必须严格遵循的执行流程（一步都不能跳过）：**

### 步骤 1：现有上下文收集与意图对齐（必须第一步）
1. **寻找历史文档**：全局搜索项目根目录及 `docs/` 目录下的 `README.md`、`DESIGN.md` 或其他 Markdown 文档。
2. **读取核心意图**：如果存在旧文档，仔细阅读其中的“背景”、“目的”或“快速开始”，理解作者造这个轮子的**初衷**和**核心解决的问题**。
3. **输出提示**：告知用户“已找到现有文档 XXX，将在此基础上进行深度代码分析与内容补全/重构”或“未找到现有文档，将从零开始构建架构分析文档”。

### 步骤 2：核心机制与流转全景扫描（核心大脑工作）
AI 必须对整个项目进行以下深度的结构分析，提取关键元数据，**不要停留在只看 Controller/Service，要用架构师的视角看代码**：
1. **入口追踪（Starter 机制）**：
   - 扫描 `spring.factories` 或 `org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件。
   - 追踪所有的 `@AutoConfiguration` / `@Configuration` 类。
   - 分析 `@ConditionalOn...` 注解，明确组件生效的前提条件。
2. **配置属性映射（Configuration）**：
   - 寻找所有带有 `@ConfigurationProperties` 的类。
   - 梳理出默认配置项是什么，支持哪些进阶配置。
3. **核心组件与接口梳理**：
   - 找出定义的顶层核心接口（Interfaces）和抽象类（Abstract Classes）。
   - 找出默认的核心实现类（Default Implementations）。
4. **扩展点与拦截机制（Hooks & Extensions）**：
   - 扫描自定义的 `BeanPostProcessor`、`ApplicationContextInitializer`、拦截器（Interceptors）或过滤器（Filters）。
   - 寻找标记了 `@ConditionalOnMissingBean` 的方法（这通常是留给使用者的扩展点）。

### 步骤 3：设计模式与执行链路逆向分析
1. **执行流程梳理**：从框架初始化（Init） -> 请求/任务触发（Trigger） -> 核心处理（Process） -> 结果收尾（Destroy/Return），构建一条完整的数据流转闭环链路。
2. **设计模式识别**：分析代码中使用了哪些设计模式（例如：通过 Interface+多个实现类判断是否为**策略模式**；通过 `build()` 判断是否为**建造者模式**；通过 `AbstractClass` 里的具体方法和抽象方法判断是否为**模板方法模式**；通过代理类判断是否为**责任链/代理模式**等），并记录具体应用在哪些类上。
3. **核心映射关系分析**（可选）：关注组件之间的映射关系（如：1:1:多 映射模型），这是理解框架设计的关键。

### 步骤 4：生成高质量架构与使用文档
汇总前面的分析，生成（或覆写更新）**分层的高质量 Markdown 文档**。
**⚠️ 核心要求：为了保证文档质量，必须【逐个生成】并写入文件，每写完一个文件都要停下来询问用户。**

文档可以参照下结构组织：
```txt
docs/
├── architecture/              # 架构设计文档
│   ├── DESIGN_DOC.md         # 系统架构、设计原理、核心组件说明
│   └── CORE_ARCHITECTURE_MAPPING.md  # 核心组件映射关系说明
├── configuration/            # 配置文档
│   └── CONFIGURATION_GUIDE.md # 完整配置说明、使用示例、最佳实践
├── guides/                   # 使用指南
│   └── INDEX.md             # 文档分类和快速导航
|
└──  其它已有目录...
```


**各文档必须严格包含以下结构：**

#### 【文档1】 `architecture/DESIGN_DOC.md` 文档：
1. **定位与核心功能 (Overview)**
   - 一句话说明这是什么（如：基于 Redisson 的分布式锁 Starter）。
   - 解决了什么痛点？核心 Feature 列表。
2. **内部执行流转与架构设计 (Architecture & Flow)**
   - **启动与自动装配流程**：详细说明该依赖引入后，Spring 容器是如何把核心 Bean 注入的。
   - **核心执行链路**：用文字或 Mermaid 流程图描述一次核心业务的完整流转过程（从 A 类 -> B 接口 -> C 实现）。
   - **核心架构映射**（可选）：明确说明组件之间的映射关系（如：1:1:多 模型）。
   - **设计模式应用**：明确指出项目中使用了哪些设计模式，并在旁边注明对应的**核心类名**。
3. **核心组件职责说明**
   - 分层说明各层组件的职责和协作关系。
   - 扩展点设计原则和机制。
4. **架构优势与设计考量**
   - 选择从性能、安全性、兼容性、可维护性等角度分析设计优势。
   - 相关文档引用（指向配置文档和指南）。

#### 【文档2】 `architecture/CORE_ARCHITECTURE_MAPPING.md` 文档：
1. **核心映射模型可视化**
   - 用 ASCII 图表或 Mermaid 图展示核心映射关系。
   - 详细说明每个映射关系的代码位置和实现逻辑。
2. **配置示例与映射分析**
   - 用具体配置示例展示映射关系的实际应用。
   - 处理流程说明。
3. **设计优势分析**
4. **注意事项**

#### `configuration/CONFIGURATION_GUIDE.md` 文档：
1. **快速开始 (Quick Start)**
   - 依赖引入、最简配置示例、业务逻辑实现。
2. **配置属性全览**
   - 如根级配置、全局配置、方法级配置的完整说明。
3. **配置示例详解**
   - 如基础配置、全局配置、混合配置的详细示例。
4. **配置验证规则**（可选）
   - 自动验证规则和配置示例验证。
5. **业务逻辑实现指南**（可选）
6. **自定义扩展配置**（可选）
7. **高级配置场景**（可选）
8. **配置最佳实践**（可选）
   - 配置组织建议、性能优化配置、安全配置建议。
9. **故障排除**（可选）
   - 常见问题、调试建议。

#### `guides/INDEX.md` 文档：
1. **文档分类**
   - 清晰的文档分类说明。
2. **快速导航**
   - 按用户角色（新用户、架构师、高级使用者）提供导航路径。
3. **常见问题**
   - 配置问题、架构问题、扩展问题的分类解答。
4. **版本信息**
5. **贡献指南**
6. **支持与反馈**
  
### 额外约束（AI 必须严格遵守的交互规范）
1. **分析与大纲确认（第一阶段停顿）**：扫描完成后，**绝对不要直接开始写文件**。必须先在对话框中输出【架构分析简报】（包含发现的设计模式、缺陷提示、核心映射模型）和【准备生成的文档树大纲】，并询问用户："以上梳理是否符合设计意图？是否可以开始生成第一个文档 DESIGN_DOC.md？"。
2. **逐个生成与写入（第二阶段循环）**：
   - 得到用户允许后，生成 `DESIGN_DOC.md`，**实际写入文件系统**。
   - 写入完成后输出："已完成 DESIGN_DOC.md。是否继续生成 CORE_ARCHITECTURE_MAPPING.md？"
   - 依此类推，直到所有 4 个文件生成完毕。
3. **README 同步更新（最后一步）**：所有 docs 目录下的文件生成完毕后，自动检查根目录的 `README.md`，如果存在根目录的 README.md，需要同步更新文档引用，指向新的文档结构。。
4. **拒绝流水账与含糊不清**：描述扩展点时，必须给出具体的类路径和方法名；文档之间的相对路径引用（如 `../configuration/...`）必须正确无误。
5. **发现缺陷主动提示**：如果在扫描代码时发现明显的设计缺陷（如循环依赖风险、缺少配置前缀等），必须在第一阶段的【架构分析简报】中作为【架构师建议】重点标出。
